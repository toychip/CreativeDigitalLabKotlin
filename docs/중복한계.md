# 중복 처리의 한계와 트레이드오프

clientEventId + DB UNIQUE 가 막는 것 / 못 막는 것, 그리고 그 이유 정리

## 막는 것

클라이언트 cross-server 재시도로 인한 events 중복 저장

- 클라 → 서버 A 전송 → 응답 못 받음 → 같은 client_event_id 로 재시도 → 서버 B 도착
- DB UNIQUE `(session_id, client_event_id, event_type)` 가 두 번째 INSERT 차단
- 모든 서버가 같은 DB 공유 → 어느 서버가 처리하든 일관

## 못 막는 것 (분산 시스템 본질적 한계)

전제: Redis(전파) 와 MySQL(저장) 은 한 트랜잭션으로 못 묶음. 두 시스템 사이 race window 상존

**한계 ① broadcast 중복 전달** (outbox 가정)

- events INSERT → Redis PUBLISH → 다른 서버가 클라에 push 완료 → "발행함" 마킹 직전 앱 다운
- 재시작 후 outbox worker 가 "발행 기록 없음" 판단 → 재 PUBLISH → 일부 subscriber 중복 수신
- 원인: PUBLISH(Redis) 와 마킹(MySQL) 이 비원자적. 두 작업 사이 사망 윈도우 상존
- 완화(제거 불가): 수신측 (sessionId, seq) dedup.

**한계 ② INSERT 와 마킹 사이 race**

- events INSERT(커밋) → PUBLISH → 마킹(별도 트랜잭션) 사이 짧은 구간에 worker 가 돌면 "재발행 필요"로 오해 후 재발행
- 완화: worker threshold 넉넉히(예: created_at < NOW()-5s). 그래도 race window 잔존

## 우리 전략 — 4 layer

| Layer | 메커니즘 | 막는 것 |
|---|---|---|
| 1 | clientEventId + DB UNIQUE | 쓰기 중복(cross-server 재시도) |
| 2 | serverId | self-echo |
| 3 | 수신측 dedup (현재 envelope.id → 권장 sessionId+seq) | broadcast 중복 전달 |
| 4 | catch-up(seq 커서) | 유실 시 클라 회복 |
